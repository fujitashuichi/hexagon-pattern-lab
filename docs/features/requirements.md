# 要件定義書：価格・ステータス制御ユニット (Financial Control Unit)

## 1. 本ユニットの主要機能
本システムは、サービス内における「数値（価格）」と「処理状態（Status）」の不整合を排除するための、最小構成のバリデーション・コアである。

* **金額の負数・型制約 (Price Integrity)**
    * 全ての数値データが `number` 型であり、かつ `0` 以上であることをスキーマレベルで強制する。
* **処理状態の決定論的遷移 (Status Enum)**
    * `draft` (下書き), `reviewed` (確認済), `approved` (承認済) の3状態以外を物理的に遮断する。
* **優先度による処理区分 (Priority Categorization)**
    * `P0` (最優先), `P1`, `P2` の定数に基づき、処理の重み付けを型安全に行う。
* **API 入出力構造の固定 (API Contract)**
    * 特定の `endpoint` に対する `method` と、付随するパケット構造を固定し、未知のプロパティ（`additionalProperties: false`）の混入を防止する。

## 2. 扱う情報の定義と制約

### 2.1 数値・ステータス制約 (RequirementItem)
| 属性 | 型 | 値制約 |
| :--- | :--- | :--- |
| **priority** | enum | `P0`, `P1`, `P2` に限定 |
| **status** | enum | `draft`, `reviewed`, `approved` に限定 |
| **id** | string | 必須。システム識別子。 |

### 2.2 金額・通信制御 (ApiRequirement)
* **概要**: 特定の金額操作やデータ取得を行う際の、通信プロトコルの制約。
* **Method**: `string`（主に `POST`, `GET` 等を想定）
* **Payload**: `requestSchema` / `responseSchema` による構造定義（空オブジェクトによる厳密な型固定）。

## 3. 非機能制約 (Non-Functional Constraints)

* **カテゴリ: Data Integrity**
    * 定義されていない未知のプロパティが 1 つでも含まれる場合、システムは即座に当該リクエストを破棄（Reject）しなければならない。
* **カテゴリ: Reliability**
    * 金額計算やステータス変更の「前」段階で、本スキーマによる形式検証を通過していることを保証する。

## 4. 運用フロー
1. **定義**: 管理者が `P0`~`P2` の優先度と `status` を割り当てる。
2. **検証**: スキーマエンジンが、入力された数値が負数でないか、enum 値が正しいかを判定。
3. **確定**: 検証を通過したデータのみが、後続の計算ロジック（BE）や表示処理（FE）へと渡される。
